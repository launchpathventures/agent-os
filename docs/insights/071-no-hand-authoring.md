# Insight-071: No Hand Authoring — Generation Is the Creation Path

**Date:** 2026-03-23
**Trigger:** Architect session evaluating Ditto's ability to drive external applications. User statement: "No one is ever going to write process by hand."
**Layers affected:** L1 Process, L2 Agent, L4 Awareness, L5 Learning
**Status:** active

## The Insight

Hand-authoring YAML — whether integration definitions or process definitions — is a developer convenience for bootstrapping, not a user-facing creation path. Real users will never write `integrations/xero.yaml` or `processes/invoice-follow-up.yaml`. Both must be generated by meta-processes running on the engine.

This applies symmetrically to two creation problems:

1. **Integration generation** — Given an API spec (OpenAPI, GraphQL schema), documentation, or source code, a meta-process produces `integrations/*.yaml` with tool definitions, auth configuration, and protocol selection. For apps Ditto built, the API surface is already known from the dev pipeline.

2. **Process generation** — Given a user's goal, industry standards (APQC), connected data evidence, and available integrations, a meta-process produces `processes/*.yaml` with steps, tools, quality criteria, and governance. The process-analyst system agent and Analyze mode are already designed for this (ADR-006, ADR-008, ADR-010) but have no equivalent on the integration side.

The integration registry and process loader are correct *runtime* abstractions — YAML files loaded, validated, and executed. But the *creation path* for both must be automated. The YAML is the intermediate representation, not the user interface.

## Implications

1. **Integration generation is a missing meta-process.** The architecture has process discovery (process-analyst, Analyze mode) but no integration discovery. A system agent that ingests API specs and generates integration YAMLs is needed. This is the prerequisite for Scenario 1 (driving platforms Ditto didn't build) and Scenario 3 (driving platforms Ditto built).

2. **Build → Integrate is a lifecycle step.** When Ditto's dev pipeline builds an application, post-build should automatically generate integration definitions from the code's API surface. Ditto built it, Ditto knows the routes, Ditto should be able to operate it. This closes the Build → Deploy → Integrate → Operate loop.

3. **Phase 6 runtime is prerequisite, not the goal.** Briefs 024-025-035-036 build the runtime foundation (registry, handlers, vault, I/O). The generation layer sits on top — it produces the YAML that the runtime consumes. Phase 6 completes the runtime; the next phase must address generation.

4. **Process and integration generation are coupled.** You can't generate a useful process without knowing what integrations are available. You can't prioritize which integrations to generate without knowing what processes the user needs. The two meta-processes must be aware of each other.

5. **Three input paths for integration generation:**
   - **API spec** (OpenAPI/GraphQL) — most SaaS platforms publish these. Highest fidelity.
   - **Source code** — for apps Ditto or another agent built. Parse API routes → generate tools.
   - **Documentation/examples** — fallback when no spec exists. LLM reads docs, generates tools.

## Where It Should Land

- **Architecture.md** — Layer 2 integration section needs a "creation path" subsection alongside the existing runtime description
- **ADR-005** — needs a follow-up section on integration generation (currently silent on how integrations are *created*)
- **Roadmap** — post-Phase 6 phase should be "Integration + Process Generation" rather than pure Awareness
- **Brief** — parent brief for the generation meta-processes, with sub-briefs for API-spec ingestion, code-based discovery, and process-integration coupling
